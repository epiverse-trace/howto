## Project identity

Epiverse-TRACE how-to guide website: recipe-style, standalone guides for outbreak analytics using R. Rendered by Quarto, deployed to GitHub Pages via GitHub Actions. License: CC BY 4.0.

## Repository layout

```         
analyses/<task>/        # One .qmd per guide, grouped by epidemiological task
internal/sandbox/       # Draft / experimental .qmd files — excluded from site render
data/                   # Raw input data (minimal; generally not committed)
docs/                   # Generated website — gitignored, never edited manually
renv/                   # Package environment (renv)
```

Task subdirectories in `analyses/`: `describe_cases`, `quantify_transmission`, `reconstruct_transmission`, `forecast_cases`, `estimate_severity`, `simulate_transmission`, `read_case`, `read_intervention`, `read_population`, `compare_intervention`, `compare_economic`

## Quarto document structure

### YAML frontmatter

``` yaml
---
title: "Active-verb phrase describing the goal"
format:
  html:
    code-link: true
editor: source
editor_options:
  chunk_output_type: console
date: last-modified
toc: true
toc_float: true
author:
  - name: "Andree Valle-Campos"
    orcid: "0000-0002-7779-481X"
---
```

**Title convention**: direct active-verb phrase, not a question. Good: "Estimate a delay-adjusted age-stratified case fatality risk" Avoid: "How to estimate...?"

### Required sections (in order)

1.  `## What do we have?` — bullet list of inputs (datasets, delay parameters, etc.)
2.  `## Steps in code` — single fenced R chunk with `#| warning: false`
3.  `## Steps in detail` — prose explanation of each package's role
4.  `## Related` — links to explanations or related guides

------------------------------------------------------------------------

## Code conventions (inside .qmd chunks)

-   Load packages first with bare `library()` calls; tidyverse loaded as meta-package
-   Use `package::function()` notation for all non-tidyverse, non-base-R calls
-   Pipe style: `%>%` with one operation per line; terminate long chains with `identity()`
-   `set.seed(33)` placed immediately before any stochastic operation
-   Preserve commented-out exploratory alternatives as documentation of what was tried
-   `glimpse()` / `head()` calls may be left in for readability
-   `#'` for prose reasoning/teaching notes; `#` for section labels and brief inline notes

## Epiverse-TRACE package pipeline

Standard data flow:

```         
Raw data → cleanepi / linelist → incidence2::incidence() → complete_dates()
  → cfr::prepare_data() / modeling packages → output
```

Key packages to prefer: `cleanepi`, `linelist`, `incidence2`, `epiparameter`, `cfr`, `EpiEstim`, `EpiNow2`, `finalsize`, `epichains`, `epidemics`, `epitrix`, `socialmixr`

## Adding a new guide

1.  Create `.qmd` in the matching `analyses/<task>/` subdirectory
2.  Use the YAML frontmatter template above (active-verb title, full author block)
3.  Follow the four-section structure
4.  If experimental, start in `internal/sandbox/` — it is excluded from site render by `_quarto.yml`
5.  Never edit anything in `docs/` — it is auto-generated by GitHub Actions

## Rendering locally

**Quarto site** — from the R console:

```r
quarto::quarto_render()        # render full website to docs/
quarto::quarto_preview()       # live-reload preview server
```

**README.Rmd** — pandoc must be reachable. Easiest path is the R console inside RStudio (pandoc is bundled). From a standalone terminal, set the path first:

```r
Sys.setenv(RSTUDIO_PANDOC = "C:/Program Files/RStudio/resources/app/bin/quarto/bin/tools")
rmarkdown::render("README.Rmd")
```

The pre-commit hook will block a commit if `README.md` is out of date with `README.Rmd`, so re-knit before committing changes to `README.Rmd`.

## Monthly maintenance

Run when asked to "run monthly maintenance for this repo". Follow these steps in order.

### Step 1 — Check upstream signals

Query the three upstream tutorial repos for recently merged "Update N packages" PRs (the monthly dependency update PRs created by their automated `update-cache.yaml` workflow):

```bash
for repo in tutorials-early tutorials-middle tutorials-late; do
  echo "=== epiverse-trace/$repo ==="
  gh api "repos/epiverse-trace/$repo/pulls?state=closed&per_page=20" \
    --jq '.[] | select(.merged_at != null) | select(.title | test("Update [0-9]+ packages")) | {title: .title, merged_at: .merged_at, url: .html_url}'
done
```

If no such PRs merged in the past 60 days, report that and stop — no maintenance needed yet.

### Step 2 — Create a maintenance branch

```bash
git checkout -b "maintenance/YYYY-MM"
```

Replace `YYYY-MM` with the current year and month.

### Step 3 — Update environment

**Run these commands locally in your R console** (Claude cannot execute R directly):

```r
renv::update()    # update all out-of-date packages in renv.lock
renv::upgrade()   # upgrade renv itself + regenerates renv/activate.R
renv::snapshot()  # save updated state
```

**EpiEstim special case** — if EpiEstim was updated, reinstall it via `renv::install()` (not `install.packages()` or `remotes::install_github()`), then snapshot again:

```r
renv::install("mrc-ide/EpiEstim")
renv::snapshot()
```

Why: `install.packages()` from r-universe records `Source: Repository`, causing CI failures — `renv::restore()` then uses `install.packages()` which cannot resolve the GitHub-only `incidence2` dev dependency. `renv::install()` records `Source: GitHub` so CI installs via remotes instead.

### Step 4 — Render check

Render each `.qmd` in `analyses/` individually (not the full site) to catch per-guide failures:

```bash
quarto render analyses/<task>/<file>.qmd --no-cache 2>&1
```

Collect every file that fails and its error message. After all renders complete, also grep each output for warnings:

```bash
grep -i "warning" <output>
```

Group warnings by guide and carry them into the Step 7 report — warnings signal deprecated APIs that will become errors in future updates.

### Step 5 — Fix failing guides

For each failing guide, read the `.qmd` and its error, then apply the appropriate action:

| Situation | Action |
|---|---|
| Renamed argument | Auto: edit the `.qmd` argument |
| Renamed function | Auto: edit the `.qmd` call |
| Changed output column name | Auto: edit downstream pipe step |
| New required argument needing domain choice | Report only |
| Model interface overhaul | Report only |

### Step 6 — Commit (isolated PR per CONTRIBUTING.md)

Stage only: `renv.lock`, `renv/activate.R` (regenerated by `renv::upgrade()`), and any fixed `.qmd` files. Commit message format:

```
chore: monthly maintenance YYYY-MM — update N packages
```

Do NOT stage data files or anything under `docs/`.

### Step 7 — Print maintenance report to conversation

Summarise in the conversation (no file needed):
- Which upstream tutorial PRs triggered the update, with links
- Which packages were updated (count + names where notable)
- Which guides were fixed automatically and what changed
- Which guides still fail and why (for human follow-up)
- Which guides emitted warnings and what they said (early signal of future breakage)

### Step 8 — Push and open PR

```bash
git push -u origin maintenance/YYYY-MM
gh pr create --title "chore: monthly maintenance YYYY-MM — update N packages" \
  --base main
```
